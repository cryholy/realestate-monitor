# 부동산 실거래가·전월세 모니터링 시스템 설계

작성일: 2026-05-04
대상 프로젝트: `/Users/joel/Claude/labs/realestate_monitor/`

---

## 1. 배경 & 목적

사용자(부동산 실수요·갭 분석 관점)는 관심 단지의 매매가가 본인 예산(20억 미만)에 도달하는 시점을 빠르게 포착하고자 한다. 기존 서비스(호갱노노·아실)는 "변동 발생 시 알림"까지만 제공하고, **목표가 도달(threshold alert)** 형태는 지원하지 않는다. 또한 매매 의사결정에는 동일 단지·평형의 전세 시세 정보(갭 = 매매가 − 전세가)가 함께 필요하다.

본 시스템은 국토교통부 실거래가 공개 API를 활용해 매일 2회 자동 점검하고, 조건 충족 시 텔레그램으로 갭 정보까지 포함한 알림을 발송한다.

### 2.2 알림 조건

- **트리거**: 매매 거래 신규 등록분이 평형·임계값 조건 충족 시.
- **전월세**: 별도 알림 트리거 아님. 매매 알림 발생 시 직전 90일 전세 시세를 컨텍스트로 첨부.
- **갭 계산**: 매매가 − 직전 90일 전세 거래 보증금 중위값.
- **순수 전세만 사용** (월세=0). 표본 부족 시 6개월로 확장하고, 그래도 부족하면 "전세 데이터 부족" 표시.

### 2.3 채널·빈도

- **채널**: 텔레그램 (워크스페이스에 이미 봇 연결됨).
- **빈도**: 매일 2회 (KST 09:00, 18:00).
- **실행 위치**: Claude 원격 에이전트(`schedule` 스킬). 노트북 OFF 상태에서도 동작.

## 4. 데이터 소스

### 4.1 API 엔드포인트

```
매매:
  https://apis.data.go.kr/1613000/RTMSDataSvcAptTradeDev/getRTMSDataSvcAptTradeDev

전월세:
  https://apis.data.go.kr/1613000/RTMSDataSvcAptRent/getRTMSDataSvcAptRent

공통 파라미터:
  serviceKey   (.env에서 로드)
  LAWD_CD      5자리 시군구 코드 (성동구 11200, 광진구 11215)
  DEAL_YMD     거래년월 YYYYMM
  numOfRows    1000
  pageNo       1 (필요 시 페이지네이션)
```

### 4.2 호출 패턴 (1회 실행당)

- 시군구 2개 × 직전 2개월 × API 2종 = **8회 호출**
- 일일 한도 1만건 대비 매우 여유.
- 직전 2개월 호출 이유: 신고시차 30일이라 "지난 달 거래가 이번 달에 신고"되는 경우가 많음.

### 4.3 응답 필드

**매매 응답** (`getRTMSDataSvcAptTradeDev`)
- 아파트, 법정동, 전용면적, 거래금액(만원), 거래년월일, 층, 건축년도, 도로명

**전월세 응답** (`getRTMSDataSvcAptRent`)
- 아파트, 법정동, 전용면적, 보증금(만원), 월세(만원), 계약일, 계약구분(신규/갱신), 층, 건축년도

## 5. 단지 매칭 전략

API는 단지 코드(kapt_code)를 제공하지 않으므로 응답 속 "아파트" 문자열로 매칭한다. 표기 흔들림(`서울숲푸르지오` vs `서울숲 푸르지오` vs `서울숲푸르지오2차`)을 처리하기 위해 다음 규칙을 적용한다.

```
정규화: 모든 공백 제거, 소문자화
매칭: name_patterns 중 하나라도 부분 포함
     AND exclude_patterns 모두 미포함
     AND 법정동 일치
```

## 6. 데이터 흐름 (매 실행 시)

```
1. config.json 로드
2. 매매·전월세 API 8회 병렬 호출
3. 매매 레코드 처리:
   ├─ 단지 매칭 + 면적 필터 + 가격 필터(< 20억)
   ├─ state.json과 비교 → 신규만 추출
   └─ 신규 매칭이 있으면 다음 단계
4. 전세 컨텍스트 계산 (매매 매칭이 있는 단지·평형만):
   ├─ 동일 단지·동일 면적, 직전 90일 전세 거래 (월세=0) 추출
   ├─ 표본 < 5건이면 180일로 확장
   └─ 보증금 중위값·최저~최고 + 갭 계산
5. 텔레그램 메시지 작성 → 발송
6. 발송 성공한 항목만 state.json에 atomic 추가 (tmp → rename)
7. logs/run-YYYY-MM-DD.log 요약 기록
```

## 7. 메시지 형식

신규 매매 매칭 1건당 카드:

```
🏠 새로운 매매 거래 (서울숲푸르지오 84㎡)

💰 매매가  19억 8,000 (15층, 2026-04-28 신고)
📊 직전 90일 전세 시세 (84㎡, 12건)
   • 중위값  12억 5,000
   • 최저~최고  11억 5,000 ~ 13억 8,000
🔻 갭 (매매 − 전세 중위값)
   약 7억 3,000

[국토부 실거래가 보기 ↗]
```

여러 건은 단지별로 묶어 한 메시지에 정리.

전세 데이터 부족 시:
```
📊 전세 데이터 부족 (직전 6개월 0건)
🔻 갭 계산 불가
```

## 8. state.json 스키마

```json
{
  "last_run": "2026-05-04T18:00:00+09:00",
  "last_error_notified_at": "2026-05-04T09:00:00+09:00",
  "alerted_sales": [
    {
      "id": "a3f7e2b1c9d8...",
      "complex_key": "seoulsupp_1",
      "deal_date": "2026-04-28",
      "alerted_at": "2026-05-01T09:00:00+09:00"
    }
  ]
}
```

- `id` = sha1(아파트+법정동+면적+거래일+층+거래금액).
- `alerted_sales` 중 180일 경과 항목은 매 실행 시 자동 정리.
- `last_error_notified_at`: 운영 알림(에러) 1일 1회 제한 추적용. 현재 시각과 24시간 비교.
- 전세는 알림 트리거 없으므로 미저장.

## 9. 에러 처리 정책

| 상황 | 동작 |
|---|---|
| API 일시 장애 (5xx, timeout) | 3회 지수 백오프 (1s, 3s, 10s) → 실패 시 error.log + 다음 실행에서 재시도 |
| API 한도 초과 (응답 코드 22) | 로그만 남기고 정상 종료 (다음날 자동 회복) |
| API 인증 실패 (코드 30) | error.log + 본인 텔레그램에 운영 알림 1회 |
| XML 파싱 실패 | 해당 응답만 스킵, 다른 시군구/월은 계속 진행 |
| 텔레그램 발송 실패 | state 미갱신 → 다음 실행에서 재발송 |
| 모든 외부 호출 실패 | 본인 텔레그램에 "운영 알림" 1일 1회 제한으로 발송 |

발송-갱신 atomicity 핵심:
1. 텔레그램 발송 성공 확인
2. state 항목 추가
3. tmp 파일에 쓰기
4. rename으로 원자적 교체

## 10. 로깅

- `logs/run-YYYY-MM-DD.log`: 각 실행 요약 (호출 N건, 매칭 M건, 발송 K건, 소요시간).
- `logs/error.log`: 스택 트레이스 + 컨텍스트.
- 일자별 파일로 누적, 90일 경과분 자동 삭제.

## 15. 운영 후 개선 트리거

`friction-log` 스킬에 다음 신호가 누적되면 개선:
- "전세 컨텍스트가 부족하다" → lookback 기간 확장
- "단지 추가하고 싶다" → config.json 직접 편집
